# Практическое задание №1: Мини веб-служба и конвейер обработки запросов

Учебный проект, демонстрирующий прохождение HTTP-запроса от входа в приложение до формирования ответа через строго заданный конвейер (middleware). Предметная область: управление пользователями (хранение в памяти).

## 🛠 Инструкция запуска и проверки

**1. Запуск сервера:**
Убедитесь, что у вас установлен Go. В корневой папке проекта выполните:
```bash
go mod tidy
go run main.go
```
Сервер запустится на `127.0.0.1:8080`.

**2. Запуск автоматизированных проверок (E2E-тесты):**
Не останавливая сервер, откройте новый терминал и запустите тесты, которые покроют позитивные и негативные сценарии (ошибки валидации 400, не найденные элементы 404, успешное создание 201):
```bash
go test ./test -v
```

**3. Ручная проверка (примеры cURL):**
- Создание пользователя (POST):
  `curl -X POST http://127.0.0.1:8080/api/users -H "Content-Type: application/json" -d "{\"email\":\"test@mail.ru\",\"password\":\"secret123\",\"name\":\"Ivan\",\"age\":20}"`
- Ошибка валидации — отрицательный возраст (POST):
  `curl -X POST http://127.0.0.1:8080/api/users -H "Content-Type: application/json" -d "{\"email\":\"test@mail.ru\",\"password\":\"sec\",\"name\":\"\",\"age\":-5}"`

## 🏛 Архитектурное пояснение: Почему это каркас, а не набор функций

Реализованное решение (связка роутера `gorilla/mux` и цепочки Middleware) представляет собой **каркас (мини-фреймворк)**. Главное отличие каркаса от библиотеки или набора разрозненных функций заключается в **инверсии контроля (Inversion of Control)** и строгом управлении жизненным циклом запроса.

Даже не запуская проект, архитектурную идею можно проследить по следующим аспектам:

### 1. Контроль над конвейером обработки (Pipeline)
Запрос не попадает в бизнес-логику напрямую. Он проходит сквозь "матрешку" из промежуточных обработчиков (Middleware): 
`RequestID -> Recovery -> Logging -> Timing -> Router -> Handler`. 
Прикладной программист, пишущий обработчик (Handler), не вызывает функции логирования или замеров времени. Каркас сам вызывает прикладной код в нужный момент, оборачивая его инфраструктурной логикой. Если в бизнес-логике произойдет критическая ошибка (паника), слой `Recovery` перехватит её до падения всего сервера.

### 2. Сквозная передача контекста
Каркас берет на себя ответственность за идентификацию запросов. На самом первом этапе (`RequestIDMiddleware`) генерируется уникальный ID и помещается в `context.Context` запроса. Бизнес-обработчики или единый шлюз вывода ошибок (`writeError`) извлекают этот ID и прикрепляют к ответу. Это избавляет разработчика от необходимости прокидывать ID аргументом через все функции слоев Service и Repository.

### 3. Декларативная маршрутизация как точка расширения
Роутер выступает контрактом. Мы не пишем спагетти-код с `if r.URL.Path == ...` и ручным парсингом параметров. Мы декларативно регистрируем точки расширения: `router.HandleFunc("/api/users/{id}", ...).Methods("GET")`. Каркас сам берет на себя разбор URL, валидацию HTTP-метода и извлечение `id`, передавая в Handler уже готовую среду для работы.

### 4. Единый формат ответов и ошибок
Вся обработка отказов замыкается на единой функции `writeError`, которая гарантирует соблюдение согласованного контракта (JSON с полями `error`, `code` и `request_id`). Независимо от того, где произошла ошибка — при парсинге JSON на входе или при проверке бизнес-правил в Service — клиент всегда получит стандартизированный ответ. Бизнес-слой возвращает обычный `error`, а HTTP-слой каркаса решает, как его правильно упаковать.

Таким образом, каркас диктует *форму* и *поток управления*, а прикладному разработчику остается лишь заполнять *точки расширения* (конкретные хендлеры и сервисы), не заботясь об инфраструктурной обвязке.
